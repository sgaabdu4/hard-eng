Single feature workflow

Plan, Implement, Verify, Ship, Learn

Run one /he:* command per stage. Start each stage in a fresh thread with the handover prompt from the prior receipt: worktree, he-state.json, stage, blockers, artifacts, and next command. The stage skill handles supporting skills, state updates, validation, receipts, and parallel subagents when work can safely split.

1. PlanDecide owner, proof, risk, and readiness.
2. ImplementChange the canonical owner.
3. VerifyProve behavior and loop on blockers.
4. ShipRun the final gate and CI follow-through.
5. LearnAdd a durable guard when needed.

Automatic Work

  • Loads the required specialist skills for the touched area.
  • Updates he-state.json before and after each internal step.
  • Records subStages[]; every required sub-stage is done or skipped with reason/evidence, and non-skippable gates must be done.
  • Records entryGate for stages 2-5 so a stage cannot start without the previous PASS.
  • Records PRODUCT.md, DESIGN.md, and token/design-system owner paths before implementation readiness.
  • Records planReadiness for Grill Me, no-guesswork alignment, UI review, Lavish decision proof, artifact status, full visible question text, and approval proof.
  • Records findings with an owner repair stage and guardrails with command, status, evidence, and push-blocking status.
  • Uses parallel subagents on gpt-5.5 when tasks are independent; evals use gpt-5.4-mini.
  • Validates state before any ready-yes handoff.
  • Ends with a compact receipt: stage, state path, decision, owner/proof, artifacts, blocker, next, handover prompt.

Guardrails

  • /he:implement starts only after /he:plan is PASS.
  • Stages 2-5 require entryGate.decision = PASS from the prior stage.
  • Non-skippable sub-stages include state validation, owner read/change, tests, quality gates, no-mistakes, durable-owner, and proof.
  • /he:ship starts only after verification is clean and work is committed.
  • State validation blocks false ready-yes handoffs.
  • Push dry-runs count only after project hooks are active and push-blocking guardrails pass.
  • SSOT scanner guardrails keep duplicated commands, scanner owners, colors, and policy concepts tied to source files.
  • Verify failures loop back through /he:implement; proof reruns after each fix.
  • Parallel subagents are used only for independent work and merge back through the active he-* stage.

Failure Loops

  • /he:plan stays in planning until missing owner, scope, proof, risk, or Grill Me alignment is resolved.
  • /he:implement loops in implementation unless owner or scope changed, then it returns to planning.
  • /he:verify sends fixes through implementation, then reruns only affected proof.
  • /he:ship uses the no-mistakes loop; code findings go to implementation, proof gaps to verify, gate evidence stays in ship.
  • /he:learn stays in learning until the durable guard owner exists and passes.
