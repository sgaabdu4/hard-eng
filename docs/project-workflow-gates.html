Single feature workflow

Plan, Implement, Verify, Ship, Learn

Run one /he:* command per stage. Start each stage in a fresh thread with the handover prompt from the prior receipt: worktree, he-state.json, stage, blockers, artifacts, and next command. The stage skill handles supporting skills, state updates, validation, receipts, and parallel subagents when work can safely split.

1. PlanDecide owner, proof, risk, and readiness.
2. ImplementChange the canonical owner after TDD proof, with SSOT owner reuse first.
3. VerifyProve behavior and loop on blockers.
4. ShipRun the final gate, CI, and loop-complete currentness proof.
5. LearnAdd a durable guard when needed.

Automatic Work

  • Loads the required specialist skills for the touched area.
  • Updates he-state.json before and after each internal step.
  • Records subStages[]; every required sub-stage is done or skipped with reason/evidence, and non-skippable gates must be done.
  • Records entryGate for stages 2-5 so a stage cannot start without the previous PASS.
  • Records PRODUCT.md, DESIGN.md, and token/design-system owner paths before implementation readiness.
  • Records planReadiness for Grill Me readiness metadata, no-guesswork alignment, explicit user-approved Grill Me skip evidence when needed, accepted UI review, saved UI review receipt, artifact status, the current visible asked question when unresolved, and approval proof; it does not duplicate the Grill Me Q/A ledger, and every ready-yes handoff preserves and revalidates it.
  • Records findings with an owner repair stage and guardrails with command, status, evidence, and push-blocking status. Repeated misses or future guards become learning/process findings. User-caught workflow/process misses recorded in state need matching repeatMisses[] or he-learn evidence before ready. Implement, Verify, and Ship also record guardrailInventory.touchedStacks[] and guardrailInventory.requiredGuardrails[] for normalized touched-stack guards. Risky E2E work records approvalBoundaries[], and repeated user-caught misses record repeatMisses[].
  • Uses parallel subagents on gpt-5.5 when tasks are independent; evals use gpt-5.6-luna. agentWork[] records status, progress, last-progress time, recovery prompt, required reason, and evidence.
  • Validates state before any ready-yes handoff.
  • Ends with a compact receipt: stage, state path, decision, owner/proof, artifacts, blocker, next, handover prompt.

Guardrails

  • /he:implement starts only after /he:plan is PASS.
  • /he:implement requires ssot-owner-reuse before test-first-proof, test-first-proof before owner change, green implementation-proof after owner change, and implementation-ui-screenshots after proof for UI-touched work.
  • Stages 2-5 require entryGate.decision = PASS from the prior stage.
  • Non-skippable sub-stages include state validation, owner read/test-first/owner-change, tests, quality gates, no-mistakes, PR review threads, durable-owner, and proof. SSOT owner reuse is also non-skippable before test-first.
  • /he:ship starts only after verification is clean and work is committed, and loop-complete requires final head and clean-worktree currentness evidence.
  • Loop-complete is blocked while open learning/process findings still route to /he:learn.
  • State validation blocks false ready-yes handoffs.
  • CONCERNS and FAIL cannot claim implementation is ready.
  • Push dry-runs count only after project hooks are active and push-blocking guardrails pass.
  • SSOT scanner guardrails keep duplicated commands, scanner owners, colors, and policy concepts tied to source files.
  • Verify failures loop back through /he:implement; proof reruns after each fix.
  • Parallel subagents are used only for independent work and merge back through the active he-* stage. Planned, running, stalled, failed, or blocked delegated work prevents a ready handoff.

Failure Loops

  • /he:plan stays in planning until missing owner, scope, proof, risk, or Grill Me alignment is resolved.
  • /he:implement loops in implementation unless owner or scope changed, then it returns to planning.
  • /he:verify sends fixes through implementation, then reruns only affected proof.
  • /he:ship uses the no-mistakes loop; code findings go to implementation, proof gaps to verify, PR review comments are closed before loop-complete, and gate evidence stays in ship.
  • /he:learn stays in learning until the durable guard owner exists and passes.
